Setting Up the SDAM Environment


This topic describes how to setup the Shared Data Access Manager (SDAM) environment.

Overview

 

Starting V5-6R2022, the legacy CORBA architecture that uses Orbix implementation and the vault can be replaced with Shared Data Access Manager (SDAM) architecture (HTTP based) for the client server communication. File Collaboration Server can be used instead of the ENOVIA V5 vault for document related transactions. 
SDAM is an in-house technology developed for communication between client and server that uses http protocol for communication.
File Collaboration Server (FCS) is a server that can be configured to handle file check-in and check-out operations when Live Collaboration is made available from the Web. A file collaboration server URL is enabled for each store or location with files that can be accessed. For more information, see Setting Up File Collaboration Server.
Either you can work with the Orbix middleware and ENOVIA vault or with Shared Data Access Manager and File Collaboration Server.
 

The following is the updated architecture to work with the SDAM and FCS mode.
Note: The SDAM and FCS mode is only supported on WINDOWS and it is not supported on AIX.
 


Before you Begin

 

  • Install the ENOVIA V5 Server, using the latest ENOVIA V5 installer. For more information, see Installing Version 5 Products on Windows.
  • Ensure that the table-space and table-owner used for DB setup during installation will be required for the FCS metadata setup.

Setup SDAM Server

 

The following steps are required to setup and start the SDAM server:

  • Start SDAM server.
  • Deploy and start SDAM Middleware.
  • Enable SDAM on client.

Start SDAM Server

 

This section describes how to start the SDAM server.
 
  • Execute “runSDAMV5Server.bat” to start the SDAM Server Manager.
  • “runSDAMV5Server.bat” file is available at following installation path “<CATInstallPath>\code\bin”.
  • “runSDAMV5Server.bat” will start the SDAM Server Manager and by default it will listen for the requests on “8025” port and will support 500 client connections.
  • SDAM Server Manager will internally use “runSDAMV5Child.bat” for starting the SDAM User Process.
  • The “runSDAMV5Server.bat” file consists of the following variables:
  1. SDAM_PORT_LISTEN: This variable defines the port for the SDAM Server Manager. Default value for the variable is “8025”. You can modify this value to make SDAM Server Manager to listen on some other port. (If modified, update the corresponding change in middleware webapp’s Web.xml)
  2. SDAM_BASE: This variable defines the port on which first SDAM User Process is started. By default, the value of the variable is “8026”. You can also modify this value if required.
  3. SDAM_RANGE: This variable defines the number of users allowed to connect simultaneously. By default, the value of the variable is “500”. You can also modify this value if required.

Deploy and Start SDAM Middleware

 

This section describes how to deploy and start the SDAM Middleware.
 
  • The SDAM Middleware is a Java web application to which clients such as LCA, VPM Navigator connect to. This application manages their communication with SDAM Server Manager and SDAM User Process.
  • Deploy it in the supported Java web container (TomEE - The path can be referred to as <SDAM middleware_Install_DIR>) and start the web container. Now the middleware is up and can accept the user connection requests.
  •  You can find WAR file “V5SDAMClientServ.war” of the middleware at the following path: “<CATInstallPath>\win_b64\resources\WebAccess”.
  • By default, the middleware is configured to connect to the SDAM Server Manager on default port configured for it in “runSDAMV5Server.bat”.
  • After installation, if you modify SDAM_PORT_LISTEN port as described in the Start SDAM Server Manager section, you must modify the respective values in “Web.xml” in the deployed web application as well.

Enable SDAM on Client

 

You must explicitly enable the SDAM environment on the LCA and VPM Navigator using the SDAMInfra flag before launching the client.

Flag Name Usage
SDAMInfra For LCA:
catstart –run “enovia –extra{ -DSDAMInfra=TRUE }extra”

For VPM Navigator:
set JAVAOPTIONS=-DSDAMInfra=TRUE
set SDAMINFRA=TRUE

Enable SDAM and FCS on ENOVIA V5 Server

 

Activate the following flags to enable in the SDAM environment to work with FCS.
 
Flag Name Usage
ENOVIA_FCSInfra Set the variable ENOVIA_FCSInfra=TRUE in the server side CATEnv file.
ENOVIA_FCSMetaServ This flag helps you to login to SDAM and FCS server in batch mode. This is not a mandatory flag for running interactive scenarios in LCA, VPM Navigator, and Interop.

The batch mode login is performed during the executing programs such as ENOVIAFileIntrospection, CATDMUUtility, ExtractForSite, and ApplyPackage, which does accept user credentials while logging in to the ENOVIA server.

This flag must be set on the server terminal or command prompt before executing the executables.

This flag has to be activated for the ENOVIA_FCSInfra on the server terminal console to use the SDAM and FCS functionality.

The value for the flag must to set as ENOVIA_FCSMetaServ=<SDAM Middleware Host Name>:<SDAM Middleware Port number>

Note: If an executable is launched in another shell or bat, it is  mandatory to define these flags (ENOVIA_FCSInfra, ENOVIA_FCSMetaServ) in the shell or bat file.

For example, During File Introspection, CATDMUUtility.bat is executed internally, hence it is mandatory to set these flags related to SDAM and FCS within CATDMUUtility.bat.
INTROSPECTION_DAEMON_USERID=<username>
INTROSPECTION_DAEMON_USERPWD=<password>
INTROSPECTION_DAEMON_USERROLE=<userrole> 		For SDAM and FCS mode, the authentication is required for user to access file collaboration server through the SDAM middleware. For authentication to work seamlessly, the user and password combination must be provided by activating the login related flags at server side for running any batches, which will interact with FCS.
