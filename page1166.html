Migrating Vault Data to File Collaboration Server

A "FCSMigration" batch tool is provided to migrate the existing data from ENOVIA vault to File Collaboration Server (FCS).

The migration tool can only be run by a user assigned with Administrator privileges and access rights to update various ENOVIA V5 tables (VPMDocFile, V514TechPackVersion, VPMNavRep_VaultItem, VPMSpacemapProperty, and VPMContext) and the new FCS tables.

This tool manages the migration seamlessly by mapping the ENOVIA V5 vault metadata and the FCS metadata so that you can start using FCS after the migration.

Before running the “FCSMigration” batch tool, update the Config.xml and dbinfo.properties files that are located at <installdir>\<OS>\code\bin

When the ENOVIA vault data is migrated to File Collaboration Server, you cannot use the same migrated data in the ENOVIA vault.
 

Updating the Configuration File

Edit the Config.xml file with the following installation parameters:

The configuration file provides the details about the mapping of the vault server and the vault location to the new FCS store.

  • properties - Root node to define the properties of mapping.
  • vaultmapping - Node representing each mapping between vault and FCS.
  • vaultserver - Name of the vault server.
  • vaultloc - Absolute path of the vault server where physical files are present.
  • fcsserver - Name of the FCS store for the vault server which need to be migrated.
 

Updating the dbinfo.properties File

Edit the dbinfo.properties file with the following installation parameters:

For Oracle:
jdbcurl=jdbc:oracle:thin:@<SDAMServerInstallationMachine>:<ListenPort>:<DBNAME>
user=<fcs_admin>
password=<fcs_admin_password>
DB=oracle

For DB2:
jdbcurl=jdbc:db2://<SDAMServerInstallationMachine>:<TCP Port for DB2 service>/<DB Name>
user=<fcs_admin>
password=<fcs_admin_password>
DB=DB2 or db2 or ibm
 

Running the Migration Tool

 

The “FCSMigration” batch file is located at <installdir>\<OS>\code\bin. Run the migration tool providing certain parameters like the following:

In Simul Mode:

FCSMigration.bat -StartDate <DD-MMM-YYYY> -EndDate <DD-MMM-YYYY> -OutDir "<Absolute Path to output directory>" -InputFile "<Absolute path to the Config.xml file>" -simul

The output shows the number of documents that will be processed if the tool is run with the same input parameters in the non simul mode.

In Actual or Non-simul Mode:

FCSMigration.bat -StartDate <DD-MMM-YYYY> -EndDate <DD-MMM-YYYY> -OutDir "<Absolute Path to output directory>" -InputFile "<Absolute path to the Config.xml file>" 

The output file shows the status of the documents that are migrated to FCS.

For example, 

FCSMigration.bat -StartDate 03-MAR-2020 -EndDate 04-MAR-2021 -OutDir "C:\temp" -InputFile "C:\temp\Config.xml"

The description of the parameters used while running the migration tool are as follows: 

  • -StartDate - The date in DD-MMM-YYYY format that indicates the created date of documents.
  • -EndDate - The date in DD-MMM-YYYY format that indicates the last modified date of documents.
Note: This date is inclusive that means the documents, which are created or updated on this EndDate will be considered for migration.
  • -InputFile -  Absolute path of the Config.xml file.
  • -simul - The simulation mode. Displays the number of documents that are considered for migration. This is the optional parameter.
  • -OutDir - The existing absolute path on which the output is generated.
  • -help - Displays the help.
 

Output

When you run the migration tool, the following output files are generated in the OutDir location.
  • ImportedData.xml – This file consists of the vault OIDs from the VAULTDOCUMENT table that indicates whether the attributes related to these OIDs are inserted in the FCS tables. If the status value is TRUE, the OIDs are inserted successful. If the status value is FALSE, check the migration tool traces to verify the failed OIDs. You can also check the failed OIDs in the UpdateResults.xml.

  • UpdateResults.xml – This file consists of the name of a document and the OID from the tables such as VPMDocFile, V514TechPackVersion, VPMNavRep_VaultItem, VPMSpacemapProperty, and VPMContext. If status is “SUCCESS”, the document is migrated successfully. If status is “FAILED”, the reason for the failed migration is provided in the file.
