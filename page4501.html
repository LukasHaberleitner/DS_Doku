Controller Profiles

Every robot controller has certain profiles each of which define characteristics of different aspects of the controller. By default, controllers by the name Default are defined for Motion, Tool, Accuracy and Object Profile.

Please see Device Task Definition: Workbench Description: Robot Controller Toolbar for more information on controller profiles.

Action Libraries

Action libraries consist of a sequence of activities that can be used to perform specific operations such as spot welding. The following action  libraries are provided for Kuka KRC controller:

  • KukaSpot action library
  • KukaServoGun action library
  • KukaGripper action library

Please see Device Task Definition: Workbench Description: Action Library Toolbars for more details.

Robot Parameter Data

Robot parameter data informs the translator about the user's special requirements. See Robot Parameter Map section for supported parameters.

The following procedure describes how to define device parameters.
Open the KukaStart.CATProcess file.

  1. Right-click robot KR125-2.1 in the PPR tree and select Properties. The properties dialog box opens.

  2. Click the Define other properties button.
    The Define Other Properties dialog box opens.

  3. Select the type of parameter from the list and click the New Parameter of type button.
    The newly created parameter appears in box listing the parameters (for example, in the dialog box above, ProgramFileEncoding has been added).

  4. In the Edit name and value field, enter a name and value as required.

  5. Or instead, you can skip steps 3-4 and import parameters from a .txt file using External properties. The .txt file must have the following format:

    parameter_name         parameter_value
    ....                           .....

Weld Guns

Kuka translators support servo and non-servo weld guns.

A servo weld gun is defined using Set Tool as an external axis of a robot, and used with KukaServoGun action library .

A non-servo weld gun is simply defined as robot-mount (Snap) or pedestal (Fixed TCP) tool, and not as an external axis. KukaSpot action library is used on non-servo weld guns. Refer to Spot Welding Application section for details.

Special Considerations

Kuka translators support Kuka Multi-Controller Robot Cooperation application (KIR). A system environment variable KUKA_TAG_ATTRIBUTE needs to be set.

For details see Kuka KIR Application section.

Kuka KIR Application

The following procedure describes how to create a KIR application.
Open the Kir.CATProcess file.

  1. Create tags that are attached to a part (corner_square in the example CATProcess) held by robot KR125-2.2.

  2. Create a task on robot KR125-2.1 containing operations that move to the tags.

  3. Switch to Table format in the Teach dialog box. Select the operations, then right-click and select Tag Reference > KR125-2.2.

    Kuka KIR tag attributes are generated upon setting Tag Reference to another robot.

     

Notes on the KIR Attributes

In the following scenario:

  • robot1 represents a robot that does some work (such as arc welding) on a part.
  • robot2 represents a robot that holds a part.

Attributes are set when Tag Reference is selected to robot2.

_kuka_kir_ref_frame: <robot2 name> + "_FLANGE", e.g. "KR125-2.2_FLANGE"

_kuka_kir_ref_frame_to_part_xform_x, y, z, w, p, r: These are the Cartesian values X, Y, Z, Yaw, Pitch, and Roll of the transform from robot2 flange to the part origin that robot2 holds. The values are output as BASE_DATA to the file specified by parameter "ConfigFile".

_kuka_base_to_ref_robot_base_x, y, z, w, p, r: These are the Cartesian values X, Y, Z, Yaw, Pitch, and Roll of the transform from robot1 to robot2. The values are output as MACHINE_DEF to the file specified by parameter "ConfigFile". For example:

MACHINE_DEF[2]={NAME[] "KR125-2.2",COOP_KRC_INDEX 9,PARENT[] "WORLD",ROOT {x 3000.0,y 0.0,z 0.0,a 180.0,b 0.0,c 0.0},MECH_TYPE #ROBOT,GEOMETRY[] " "}

In the case of an auxiliary device, such as a positioner instead of a robot, MECH_TYPE should be #EASYS. Setting Tag Reference to auxiliary devices is not currently supported.

For KIR applications, tasks must be downloaded in Part Coordinates. The device attribute ConfigFile must be specified, and the specified file must exist. Otherwise, an error message is displayed.

KUKA KIR download must be performed in Part Coordinates. Please re-download and select Part Coordinates.

  3. Download the task in Part Coordinates.

On upload, robot moves using base1 will have tags created and attached to part1. This eliminates the Part Coordinates option for upload. For tags not attached to any product, and the moves use Default Object Profile, the moves will be downloaded with Base[0].

External Axis – Rail

For rail applications, it is necessary to use the parameter MachineFile to point to the $machine.dat file that defines the robot-rail configuration.

This section shows the variable values in the $machine.dat file that determine the robot-rail configuration.

90-degree standard configurations

Configurations Variable Settings
A $ET1_TA1KR = {X=0, Y=0, Z=450, A=0, B=90, C=0}
$ET1_TFLA3 = {X=0, Y=0, Z=0, A=0, B=-90, C=0}
B $ET1_TA1KR = {X=0, Y=0, Z=450, A=0, B=0, C=-90}
$ET1_TFLA3 = {X=0, Y=0, Z=0, A=0, B=0, C=90}
C $ET1_TA1KR = {X=0, Y=0, Z=450, A=0, B=-90, C=0}
$ET1_TFLA3 = {X=0, Y=0, Z=0, A=0, B=90, C=0}
D $ET1_TA1KR = {X=0, Y=0, Z=450, A=0, B=0, C=90}
$ET1_TFLA3 = {X=0, Y=0, Z=0, A=0, B=0, C=-90}

Note that the Z value(450 in the example) specifies the height of the robot raised by the rail. The values of B and C determine the configurations.

For the 90-degree configurations, the tags using zero ObjectFrameProfile(such as the Default object frame profile) and base data output are relative to a coordinate system Z-offset below the robot coordinate system.

Moving Object Frame Profiles

Kuka allows you to define targets relative to a Base that is fixed on a workpiece positioner.

A Product (to use its origin) can be created and placed on a positioner where the desired Base is.

A Tag Group can then be created on the Product, and tags are created at desired locations. Each robot motion should use Default Object Profile.
 
When a Tag Group is attached to a Product that is in turn attached to a positioner, with each move that involves the positioner (and the Product), an ObjectFrameProfile is generated in the file simResult.xml. The ObjectFrameProfiles are the Product origin relative to the robot selected for download. These ObjectFrameProfiles are not useful because they change locations.
 
You can create a file with only the PART_BASE_MAP definitions and use parameter ConfigFile to specify its path and name. The array variable PART_BASE_MAP specifies Base names for each part that has a Tag Group attached to it.

STRUC PART_BASE_T CHAR PART[24], BASE[24]
DECL PART_BASE_T PART_BASE_MAP[16]
PART_BASE_MAP[1]={PART[] "KR125-2.1",BASE[] "base1"}
PART_BASE_MAP[2]={PART[] "Positioner",BASE[] "base2"}
PART_BASE_MAP[3]={PART[] "",BASE[] ""}

With the above definitions, targets attached to robot KR125-2.1 (Product Instance name) will have Base base1, and targets attached to Positioner will have Base base2.
